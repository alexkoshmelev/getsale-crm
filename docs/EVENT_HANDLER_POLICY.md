# Event Handler Error and Retry Policy

Политика обработки ошибок и повторных доставок для подписчиков RabbitMQ (exchange `events`).

---

## 1. Инфраструктура (shared/utils RabbitMQClient)

- **Retry:** при падении обработчика сообщение не ack’ится; RabbitMQ передоставляет. В коде клиента заданы `MAX_RETRIES = 3` и заголовок `x-retry-count`; после исчерпания попыток сообщение уходит в **DLQ** (dead-letter queue).
- **Очереди:** каждый сервис подписывается со своей очередью (например `messaging-service`, `automation-service`), чтобы сбой одного не блокировал других.

---

## 2. Критичные обработчики (должны быть надёжными и по возможности идемпотентными)

| Сервис | События | Действие | Политика при ошибке |
|--------|---------|----------|----------------------|
| **automation-service** | LEAD_SLA_BREACH, DEAL_SLA_BREACH, LEAD_STAGE_CHANGED, DEAL_STAGE_CHANGED, MESSAGE_RECEIVED, CONTACT_CREATED | Создание сделки, смена стадии, уведомления | Retry через RabbitMQ; при 409 (deal already exists) — skip и инкремент метрики; ошибки логировать и пробрасывать (не глотать), чтобы сработал retry/DLQ. |
| **messaging-service** | LEAD_CREATED_FROM_CAMPAIGN | attachLead (привязка лида к диалогу) | **Идемпотентно** по (conversationId, leadId). При ошибке — log.error, повторная доставка обработается как no-op. Retry/DLQ по умолчанию. |
| **campaign-service** | MESSAGE_RECEIVED, LEAD_CREATED, LEAD_STAGE_CHANGED | Обновление лидов, конвертация, ответы | Ошибки логировать; при сбоях внешних вызовов (pipeline, messaging) — retry за счёт RabbitMQ. |

**Рекомендации:** для критичных действий (создание сделки, смена стадии) обработчики должны быть идемпотентны по бизнес-ключу (leadId, dealId, ruleId + breachDate и т.п.) или явно проверять «уже обработано», чтобы повторная доставка не создавала дубликаты.

---

## 3. Best-effort обработчики (допустима потеря события)

| Сервис | События | Действие | Политика при ошибке |
|--------|---------|----------|----------------------|
| **analytics-service** | DEAL_STAGE_CHANGED, DEAL_CLOSED, MESSAGE_SENT | Запись метрик | Логировать ошибку; не пробрасывать (чтобы не уходить в retry/DLQ по каждому сбою). Потеря одного события допустима. |
| **activity-service** | Набор типов (действия пользователей) | Запись в activity feed | Аналогично — log и не пробрасывать. |
| **ai-service** | MESSAGE_RECEIVED | Генерация черновика ответа | Best-effort; при ошибке (OpenAI, и т.д.) — log, не пробрасывать. |
| **websocket-service** | MESSAGE_*, DEAL_*, AI_DRAFT_* и др. | Broadcast в реальном времени | Потеря события — клиент не увидит обновление в реальном времени; повторная доставка может привести к повторной отправке. Допустимо логировать и не пробрасывать. |

---

## 4. Идемпотентность (кратко)

- **attachLead (messaging):** по ключу (conversationId, leadId). UPDATE с условием `lead_id IS NULL OR lead_id = $1` — при повторной доставке 0 строк или 1 строка, дубликатов нет.
- **automation create deal:** при 409 от CRM (deal already exists) — считаем обработанным, skip.
- **automation move stage:** вызов к CRM/pipeline; при повторной доставке возможен повторный PATCH — желательно, чтобы API смены стадии было идемпотентно по (dealId, stageId) или (leadId, stageId).

---

## 5. Что делать при доработках

1. **Новый критичный обработчик:** сделать логику идемпотентной по бизнес-ключу; при ошибке — логировать и пробрасывать (throw), чтобы сработал retry/DLQ.
2. **Новый best-effort обработчик:** в catch — только log, не throw.
3. **Сомнения по типу:** по умолчанию считать best-effort (log, не throw); перевести в критичные только при явном требовании к надёжности.
