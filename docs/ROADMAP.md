# Дорожная карта

**Дата обновления:** 2026-04-06

Единая картина оставшихся задач и приоритетов. Обновлять после каждого крупного шага.

---

## 1. Приоритеты продукта (в порядке важности)

1. **Доверие и доступ:** восстановление пароля по email, верификация email (блокируют B2B-онбординг сильнее, чем OAuth).
2. **CRM операции:** массовые действия (bulk delete/update), импорт/экспорт CSV — снижают трение ежедневных операций.
3. **Воронка:** история переходов по стадиям, правила entry/exit, авто-переходы — качество аналитики и автоматизации.
4. **AI в сделке и кампании:** автосоздание сделки/лида из чата (правила или AI по намерению); виджеты в карточке сделки (следующий шаг, вероятность); AI-персонализация рассылок.
5. **Омниканал:** модель channels + conversations, единый timeline по контакту; каналы помимо Telegram (WhatsApp, Email, Instagram DM).

---

## 2. Что не реализовано или неполно

### Auth
- Восстановление пароля по email (SendGrid/Resend)
- Верификация email
- OAuth (Google/GitHub/Telegram)
- Account lockout, детальный audit по входам
- 2FA (TOTP + recovery codes) — реализовано в auth-service; при необходимости — доработка UX и политик

### CRM
- Массовые операции (bulk delete/update)
- Импорт/экспорт (CSV)
- Мягкое удаление (soft delete) при необходимости

### Воронка (Pipeline)
- История переходов по стадии
- Валидация правил entry/exit
- Авто-переходы по правилам (управление воронками/стадиями PUT/DELETE + UI уже сделано)

### Кампании
- Дальнейший rate limit по каналу
- AI-персонализация
- Рефакторинг толстого `campaigns.ts`

### AI
- Автосоздание сделки/лида из чата (правила или AI по намерению)
- Виджеты в карточке сделки (следующий шаг, вероятность закрытия)

### Права
- Расширение `canPermission` в CRM и других сервисах (messaging и bd-accounts уже используют `role_permissions`)

## 3. Технический долг

### Границы данных
- **Убрать SQL-путь в `MessageDb`** полностью (только internal API messaging); до этого можно держать `BD_ACCOUNTS_MESSAGE_DB_STRICT` — см. [DEPLOYMENT](operations/DEPLOYMENT.md)
- Fallback orphan при удалении BD — мониторить метрику `bd_accounts_messaging_orphan_fallback_total`

### Надёжность
- Уточнить пороги алертов `InterServiceHttp*` под реальный трафик
- Редкие GramJS `invoke` без FloodRetry (реакции в UI, фильтры диалогов) — по метрикам
- Стратегия партиционирования/архивации `messages` при росте

### Качество кода
- Централизованный `validation.ts` для **team**, **analytics**, **activity** — завершить (частично сделано)
- Слой репозитория / query-модули для тяжёлых агрегатов (список чатов, campaign participants)
- Рефакторинг толстых роутеров: `campaigns.ts` в campaign-service
- Точечный grep на оставшиеся прямые `apiClient` вызовы на фронте

### Инфраструктура
- Детальные rate limits по типу операции
- Расширение мониторинга (дашборды, покрытие всех критичных цепочек)
- E2E-тесты ключевых сценариев
- CSP и theme script

## 5. Средний и длинный срок

### Средний срок
- Unified Inbox: модель channels + conversations, единый timeline по контакту
- Детальные rate limits (чтение vs отправка, по организации)
- Мониторинг: структурированные логи, метрики, дашборды и алерты
- E2E-тесты: регистрация, вход, создание компании/контакта/сделки, подключение TG, отправка сообщения

### Длинный срок
- Омниканал (WhatsApp, Email, Instagram DM)
- Расширенная аналитика и отчёты, экспорт
- Мобильное приложение или PWA
